# Database Audit — JezSy (admin-dashboard + jezsy-mobile-app)

Date: 2026-07-20. Phase 1: audit only — no schema changes made or proposed here.

**Stack (Phase 1 discovery):** Supabase Postgres, one shared project. Both apps use `@supabase/supabase-js` directly (no ORM). Migrations are plain SQL: `supabase/migrations/` in each repo. The admin dashboard shows a completed Firebase→Supabase migration (leftover `firestore.rules`, `firebase.json`, and Firestore-era naming like `product_doc_id`). Live schema inspected directly via the Supabase API; both repos' code grepped for every table/RPC/bucket touchpoint.

## Findings, worst first

### 1. CRITICAL — `devices` table is world-writable to any signed-in customer

- **Evidence:** live RLS policy `Enable upsert for authenticated users` on `public.devices`: `FOR ALL USING (true) WITH CHECK (true)` for role `authenticated` (confirmed by Supabase security advisor `rls_policy_always_true`).
- **Consequence:** `devices` is the admin dashboard's device-approval gate (fingerprint, `status`, `failed_attempts`, `lockout_until`, `login_history`, staff emails/names). Any customer authenticated through the mobile app can read every staff device record, set any device's `status` to approved, clear lockouts, or delete rows — defeating the dashboard's device-security model entirely.
- **Severity:** breaks security. Highest priority in this audit.

### 2. CRITICAL — customers can write price columns directly (RPCs are bypassable)

- **Evidence:** live policies: `reservations` INSERT/UPDATE `WITH CHECK (customer_id = auth.uid() OR is_admin_or_owner())`; `orders` has `Users can manage their own orders` `FOR ALL USING (auth.uid() = customer_id)`; `order_items` has an equivalent `FOR ALL` via parent-order ownership. `create_reservation` and `create_order` compute prices server-side but are `SECURITY INVOKER`, so the direct PostgREST path (`POST /rest/v1/reservations` with `rental_price: 0`, or UPDATE of `orders.total_amount`) remains open.
- **Consequence:** any tampered client can create or amend reservations/orders at arbitrary prices. The UPDATE path also defeats correctly-priced rows after the fact.
- **Severity:** breaks data integrity (financial).
- **Note:** a fix is already drafted, unapplied and uncommitted, at `supabase/migrations/20260720140000_close_price_manipulation_vector.sql`. Cross-checked against admin code: the only admin write is `recordBoutiqueSale` (`src/services/productService.js:235`), which the drafted fix preserves via `is_admin_or_owner()`.

### 3. CRITICAL — three-way migration-history drift, including a version collision

The single most likely root cause of "issues that don't make sense." Three sources of truth disagree:

| Source | Contents |
|---|---|
| Mobile repo (15 files) | `20260629083800` … `20260720140000` |
| Admin repo (5 files) | `20260712000000` … `20260716000000` |
| Live ledger (17 rows) | `20260712000000` … `20260720075401` |

Specific defects:

- **3a. Version collision, different content:** mobile `20260715000000_wardrobe_setup.sql` vs live/admin `20260715000000_fix_rls_role_casing`. Any `supabase db push` from the mobile repo will consider `wardrobe_setup` already applied because the version number matches a different migration.
- **3b. Nine mobile migrations applied out-of-band:** everything from `20260629083800` through `20260716130000` exists as a file but has no row in the live ledger (they were evidently run via SQL editor/MCP). CLI tooling will try to re-apply them.
- **3c. Same content, different versions:** the four PR #3 migrations exist on disk as `202607191200/1215/1230/1300` but were recorded live as `20260719141104/141218/141630/141803`. Each is simultaneously "unapplied" (by file version) and "untracked" (by ledger version).
- **3d. Six live migrations exist in neither repo:** `add_profile_self_insert_policy`, `drop_insecure_profile_self_insert`, `restrict_staff_write_rls`, `add_products_category_fk`, `backfill_inventory_rows`, `fix_category_id_ambiguous_match`. Live schema changes with no source-controlled definition.
- **3e. One file unapplied:** `20260720140000_close_price_manipulation_vector.sql` (deliberate — awaiting sign-off).

- **Consequence:** no migration tool can currently be trusted against this project; every future schema change risks partial application or silent skips. Schema drift also guarantees `src/types/database.types.ts` goes stale unpredictably.
- **Severity:** breaks change management (and indirectly, data integrity).

### 4. HIGH — `profiles.role` is nullable free text, and it is the foundation of all RLS

- **Evidence:** live column: `role text NULL DEFAULT 'customer'`, no CHECK constraint. Every privilege helper (`is_admin_or_owner`, `is_staff_or_admin`) reads it. The admin repo's own `20260715000000_fix_rls_role_casing.sql` exists precisely because `'Admin' vs 'admin'` casing already caused an RLS outage once.
- **Consequence:** a typo'd or miscased role value silently strips (or in the worst case grants) privileges; NULL role makes a user invisible to both helpers. No database-level guard prevents recurrence of the exact bug that already happened.
- **Severity:** correctness/security, one typo away from recurrence.

### 5. HIGH — anon/authenticated can execute sensitive SECURITY DEFINER RPCs

- **Evidence (security advisor + live grants):**
  - `check_email_exists(text)` — callable by `anon`: account-enumeration oracle at `/rest/v1/rpc/check_email_exists`. (Mobile does call this pre-signup, so it is load-bearing; the exposure is the trade-off to document or rate-limit.)
  - `update_user_streak(p_user_id uuid)` — callable by `anon` and `authenticated` with an arbitrary target user: anyone can modify anyone's streak.
  - `update_staff_status(...)` — callable by `authenticated`; safe only if its body re-checks the caller's role (unverified).
  - Trigger functions (`handle_new_user`, `sync_conversation_on_message`, `approve_device`, etc.) are individually executable via RPC; they fail outside trigger context but should not be exposed.
- **Severity:** security, moderate blast radius each.

### 6. MEDIUM-HIGH — status columns are free text with inconsistent casing across the two apps

- **Evidence:** DB defaults are lowercase (`reservations.status DEFAULT 'pending'`, `payment_status DEFAULT 'pending'`, `orders.status DEFAULT 'pending'`). Mobile writes `'Pending'`/`'Completed'` (via `create_reservation`) and filters `.eq("status", "Completed")` (`src/components/TimeSlotPicker.tsx`). Admin writes **both** `status: 'Completed'` and `status: 'pending'` (`src/services/*.js`). No CHECK constraints anywhere.
- **Consequence:** equality filters silently miss rows depending on which writer created them — the classic "reservation exists but doesn't show up" bug. A defaulted row (`'pending'`) will not match mobile's `'Pending'`-based UI states.
- **Severity:** correctness bug, actively latent.

### 7. MEDIUM — two sources of truth for stock

- **Evidence:** `products.stock` (int) and `inventory` (per-size rows with `total/reserved/available`) coexist. Admin maintains both client-side (`syncProductStock`, `recordBoutiqueSale` in `productService.js` update inventory then products). Mobile reads **both** (`app/product/[id].tsx:68` reads `inventory`; catalog reads `products`). `stock_movements.change_type` CHECK only allows `manual_adjustment|restock|correction` — there is no movement type for reservations/sales, and no trigger links any of the three.
- **Consequence:** the canonical "admin shows different stock than the app" symptom. Any client crash between the two writes desyncs them permanently; nothing reconciles.
- **Severity:** correctness bug.

### 8. MEDIUM — admin calls an RPC that does not exist: `merge_message_reaction`

- **Evidence:** `src/services/communicationService.js:70` calls `supabase.rpc('merge_message_reaction', …)`; live function list has no such function. The "fallback" comment at line 76 is aspirational — the code only logs a warning.
- **Consequence:** message reactions from the dashboard silently no-op, every time. Also note `messages` has no `reactions` column live, so the feature has no storage even if the RPC existed.
- **Severity:** functional bug (silent).

### 9. MEDIUM — uniqueness gaps and duplicates

- `reservations.display_id` has **no unique constraint** (`orders.display_id` does). The RPC's check-then-insert loop is race-prone; concurrent inserts can produce duplicate customer-facing reservation numbers. 
- Duplicate identical indexes (performance advisor): `wishlists` (`wishlists_user_id_product_id_key` = `wishlists_user_product_uidx`), `orders` (`orders_display_id_key` = `orders_display_id_uidx`) — artifacts of migrations applied twice under different names, i.e. direct fallout of finding 3.
- `color_options` vs `color_list` (8 rows each): near-duplicate tables; admin uses `color_list`/`pattern_list`; nothing in either repo references `color_options` → dead table from the Firebase migration.

### 10. MEDIUM — RLS policy sprawl

- **152 "multiple permissive policies" warnings across 20 tables** (advisor): overlapping generations of policies OR together, so the broadest always wins and auditing any table requires reading 4–6 policies. Concrete examples verified live: `orders` (2 FOR ALL + 2 INSERT + dup SELECTs), `order_items` (6 policies), `reservations` (6).
- `USING (true)` UPDATE policies on `categories`, `color_list`, `pattern_list`, `products` (admin gate only in WITH CHECK) — works, but advisor-flagged and inconsistent with the owner-scoped style used elsewhere.
- 64 `auth_rls_initplan` warnings: `auth.uid()` unwrapped (not `(select auth.uid())`), re-evaluated per row on large scans.

### 11. LOW-MEDIUM — 20 unindexed foreign keys

Most consequential given query patterns: `messages.conversation_id` (chat pagination), `conversations.customer_id`, `reservations.product_id` + `customer_id` filters, `inventory.product_doc_id`, `reviews.user_id`. Full list in the performance advisor. Trivial at current row counts (≤78 rows), but chat tables grow unbounded.

### 12. LOW — vestigial rental-model schema and dead objects

- `reservations.rental_price` / `return_date`: the business is reservation-only (no return concept — confirmed 2026-07-20). `return_date` is now left NULL by the RPC but the column and misleading name remain; UI label already fixed. Renaming requires lockstep changes in both apps.
- Dead/orphaned: `create_reservations_from_cart` (revoked, uncalled), `rls_auto_enable`, `color_options` (above), `pose_guides` (0 rows, referenced by neither repo's greps), `suggested_outfits` (0 rows, no code references found), `messages.read_at` written by no one found in either repo, `conversations.unread_count` (mobile sums it for a badge but nothing ever increments it — known flagged issue awaiting admin coordination).
- Storage: four public buckets allow listing (`ar-models`, `avatars`, `chat-images`, `wardrobe-images`); leaked-password protection disabled in Auth settings.
